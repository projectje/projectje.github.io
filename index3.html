👪 Het model en de software

Op dit moment gebruiken we een spreadsheet, wat prima werkt. Voor geavanceerdere ideeen is een ander software platform nodig. Vandaar de keuze om deze pagina te hosten op GitHub.
Deze pagina fungeert als een verzamelplaats voor ideeen voor zowel het model als de software.
Het uiteindelijk droomdoel is om een stuk gezins-software te hebben dat alle gezinsprocessen herkent en daarbij kan adviseren of helpen ,liefst met zoveel mogelijk integratie naar 3rd party's en gratis op GitHub. Dit zal dus geen budget of personal finance tool zijn omdat we denken dat een gezins module veel breder is maar wel moet kunnen integreren met "budget posten" e.d. omdat het "gezin" is, moet informatie die de een veranderd, gelijk zichtbaar zijn voor de ander.

Het Transactie Model

Transactie functionaliteit op dit moment in het sheet:
Transacties worden nu geimporteerd in een sheet. Daarna worden ze gelabeled en worden er andere slimme operaties op uitgevoerd zoals het herkennen van het pasnummer bij persoon.
Er zijn totaalsheet in een oogopslag over jaren, maanden, per post of totalen direct een goed overzicht geven.
Per post is dan nog informatie zoals verloop grafiek en ruimte voor analyse, iets slimmere auto analyses zouden handig zijn
Een meer open verwerkings engine die regels e.g. vanaf github ophaalt waar ze gedeelt kunnen worden zou handig zijn
In den beginne links per post naar de pakweg top 25 vergelijkings en informatie sites over het onderwerp zouden ook handig zijn ter deling.
De Taxonomie moet open zijn: we willen aantal "by convention" groepen toevoegen, waarvan we weten dat iedereen deze gebruikt. Hier zijn veel websites over. We moeten een mapping component toevoegen die kan mappen tussen die verschillende taxonomies.
Er zijn nog veel ideeen ter verbetering maar dit is een klein stapjes vooruit ten opzicht van het huidige spreadsheet.

Het Gezins Model

Dit zou het onderdeel moeten zijn met de eenvoudige "GUI", gebruikers zouden nooit direct naar de transactie module moeten hoeven gaan. Tenzij ze dat prefereren, e.g. om langzaam erin te rollen.
We kunnen hier hard van stapel lopen maar in principe om het scopen zijn alle posten die op dit moment gelinked zijn aan "groepen transacties" kandidaat om uit te breiden tot software modules met varieerende functionaliteiten "alle". Zo zou de energie module, automatisch voorstellen moeten doen en in een volgende fase ook moeten adviseren op basis van een veelvoud aan informatie bronnen. In beginsel kan elk van deze modules een link pagina hebben naar vergelijkingssites, apps, tools, etc... Deze verzameling moet wel "open" zijn. Een bestaand bookmark project dat uit browser urls haalt en deze in pagina's omtovert kan als basis hiervoor gebruikt worden. (een github directory bevat dan louter folders met .URL ini files). Dit is software matig vrij eenvoudig te realiseren.
Zo zou een groot deel van de informatie die we nu in OneNote hebben, in het tool kunnen belanden.
Een specifiek onderdeel is de contacten die we nu in OneNote bijhouden: in grotere bedrijven zijn dit CRM systemen. Dit zou een onderdeel kunnen zijn wat ook op de shortlist staat om te voegen, omdat hier dan automatisch voor elke "post" (of de toekomstige naam) adres en contactgegevens van een suppplier toegevoegd kunnen worden.
Andere standaard onderdelen zijn de noodnummers pagina, de lange termijn lijst, de taxonomie van alle spullen die we in huis hebben en de korte termijn todo pagina.

De Software

  • Als het model er staat kan de software in elke taal geimplementeerd worden.
  • Een elektron client is een optie als eerste platform om de functionaliteit die nu in het spreadsheet zit te porteren. Omdat het op meerdere platformen tegelijkertijd te deployen valt en delen van de code herbruikbaar zijn en omdat het een eenvoudige "klik run" ervaring is. Daarnaast zijn er ontzettend veel developers met ervaring in JavaScript dus het is eenvoudig om code bij te dragen. Een C# Core client is een ander idee, ook vrij eenvoudig en snel te bouwen een een grote set van open componenten voor beiden is beschikbaar. Dus er kan snel functionaliteit worden toegevoegd.
  • Gezinnen moeten onderling informatie kunnen uitdelen, dus er moet een communicatie laag zijn die (veilig) pear to pear kan communiceren. Er zijn een aantal 3rd party programma's die interessant zijn om hier een keuze uit te maken.
  • Gebruikers die een aankoop willen doen bijvoorbeeld een stofzuiger gaan op DAT moment vergelijken. Dus een spreadsheet component (met filter) o.i.d. kan een gemeenschappelijke bij te houden lijst zijn met features en prijzen. De gebruiker spaart aan de ene kant tijd hierdoor aan de andere kant zit hij nu in de modus om te vergelijken dus kan hij net zo goed het bestaande sheet updaten en dat als bron nemen. iedereen profiteert hiervan. De data wordt dan in een github repository opgeslagen, zodat deze ook buiten het programm om geupdated, geraadpleegd kan worden.
  • Waarschijnlijk is er een server variant benodigd, als alle data in GitHub repositories wordt opgeslagen, bijvoorbeeld het delen van herkennings regels, urls, tips en data, dan zou dit gelijk geversioneerd zijn en "open".
